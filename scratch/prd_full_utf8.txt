PARTE 1: Resumen Ejecutivo y Arquitectura (PRD)
1. Visión del Producto y Objetivos
"KUYEN convierte cada lápida del Cementerio Municipal de Angol en una puerta digital hacia la historia viva de la ciudad, devolviendo dignidad a los que ya no pueden hablar y conectando a las familias anglinas con su memoria colectiva."

Problema que resuelve
El Cementerio Municipal de Angol alberga figuras históricas, loncos mapuche, educadores pioneros y fundadores cuyas tumbas se deterioran sin documentación accesible. Los familiares no tienen canal oficial para aportar fotografías o biografías, y la administración municipal carece de herramientas digitales para gestionar el patrimonio funerario.

Objetivos Estratégicos

Principios de Diseño No Negociables
Light Mode obligatorio. Los visitantes leen bajo luz solar directa. Contraste mínimo WCAG AA (4.5:1). Fondo blanco, texto negro.
Mobile-First extremo. El diseño parte del viewport de 375px. Desktop es una mejora progresiva.
Sin dependencia de JS para contenido público. Las páginas de memorial renderizan con SSG/SSR.
QR inmutables. La URL corta /q/c001 nunca cambia aunque el slug del perfil se edite.
Soberanía de datos. Todo alojado en OCI ARM propio. Ningún dato personal sale a terceros.

2. User Personas
Persona A — Visitante del Cementerio

Persona B — Familiar / Contribuyente

Persona C — Curador / Administrador Municipal

3. Diagrama Arquitectónico
KUYEN se despliega íntegramente sobre Oracle Cloud Infrastructure (OCI) con arquitectura ARM Ampere A1 Flex. Todo el stack corre sobre Docker Compose dentro de una instancia Ubuntu 22.04 LTS.


Flujo de petición QR físico
Visitante escanea código QR de la placa física.
Móvil solicita: GET https://kuyen.cl/q/c001
Next.js Middleware (edge runtime) intercepta la petición.
Consulta caché en memoria: si existe → redirect 307 inmediato.
Si no hay caché → consulta PocketBase: SELECT redirect_slug WHERE code='c001'.
Almacena en caché (TTL 5 minutos) y redirige 307 a /memorial/saavedra.
Next.js SSG page renderizada → HTML completo enviado al móvil.
Fire-and-forget: incrementa visit_count del citizen en PocketBase.

4. Modelo de Datos Base — Colecciones PocketBase
PocketBase usa SQLite como motor de base de datos. Todas las colecciones se crean desde el Admin UI de PocketBase (/_/) y se exportan como pb_schema.json para versionar en el repositorio.

Épica 4: Visualización de Árboles Genealógicos (React Flow)
Objetivo: Visualizador de árboles genealógicos interactivo embebido en las páginas de memorial, sin conflictos de scroll en móvil, nodos personalizados con foto, layout Dagre en el servidor, y cobertura Playwright de gestos táctiles.

Decisiones de Arquitectura de la Épica
Gestión del conflicto de scroll móvil: Botón "Activar mapa" que habilita la interacción del canvas solo cuando el usuario lo solicita. En reposo, el canvas es estático y el scroll de página funciona normalmente.
Layout automático: Dagre ejecutado en el servidor (Server Component) para calcular posiciones x,y antes de enviarlos al cliente. React Flow recibe posiciones precalculadas, eliminando saltos visuales.
Carga diferida: React Flow y Dagre (~380KB) se cargan con next/dynamic y ssr: false únicamente cuando el componente entra en viewport. Un citizen sin relaciones no recibe ningún bundle de React Flow.

Criterios de Aceptación (Playwright E2E)

✓  Al hacer scroll hasta el árbol aparece el skeleton de carga del canvas
✓  El árbol NO se renderiza en el HTML del servidor (ssr: false verificado)
✓  En citizen SIN relaciones, el componente de árbol no se monta

Épica 5: Crowdsourcing y Bandeja de Moderación Administrativa
Objetivo: Flujo completo de contribución ciudadana — formulario público con upload de fotos, notificaciones por email, bandeja de moderación con aprobación granular, y tests E2E del ciclo completo.

Decisiones de Arquitectura de la Épica
Server Actions para el formulario: El formulario de contribución usa Next.js Server Actions con FormData nativo. Permite subir archivos directamente al servidor con validación Zod antes de tocar PocketBase.
Bandeja de moderación en Next.js: El panel admin vive en /admin/* dentro del mismo proyecto Next.js, protegido por middleware de autenticación. No es una aplicación separada.
Email via PocketBase hooks nativos: PocketBase tiene sistema de email transaccional con templates. Se configura hook afterCreate en proposals y afterUpdate para notificaciones. Sin dependencia de SendGrid.
Aprobación selectiva de campos: El administrador puede aprobar la biografía, rechazar las fotos y pedir revisión del documento. Evita publicar contenido parcialmente revisado.
